# Stuck Checklist

5 分から 10 分ほど止まったら、アルゴリズム名を探す前にこのチェックを回す。

## 1. 固定できるものはないか

- 区間そのものではなく、左端・右端・最小値・最大値・最後に使う要素を固定できないか
- 「右にも左にも伸びる」と感じたら、どこか 1 点を固定して場合の数に落とせないか
- 区間を数える問題なら、「その要素が何回数えられるか」に変換できないか

## 2. 欲しい量を取り違えていないか

- 典型 DP が数える「通り数」を追っているが、実際に欲しいのは「成立するか」「最小の成立境界」ではないか
- 左端や右端を固定したとき、一度成立したら以後ずっと成立する単調性はないか
- 各固定点について「初めて成立する位置」さえ分かれば、一括で数えられないか

## 3. 判定を先に言語化できないか

- まず愚直に「この区間 / 状態が成立するか」を日本語で 2 行くらいで書けるか
- その判定で必要なクエリは何か
- `次に現れる位置`, `直前 / 直後の障害物`, `最初に超える時刻` のようなクエリに分解できたら、前計算や表引きで高速化できないか
- 判定が見えた直後に、「これを DP / 尺取りにするには」と型の名前へ飛んでいないか
- 尺取りを疑うなら、右を伸ばす更新だけでなく、左を 1 つ進めたときの更新も軽いか確認したか

## 4. 寄与で見られないか

- 1 個の要素・辺・頂点が答えに何回足されるかだけを数えられないか
- 総和が重いなら、`sum over interval` ではなく `sum over element contribution` にできないか
- シグマが見えたら、まず紙に展開してから高速化を考える

## 5. 向きを変えると簡単にならないか

- 始点から終点ではなく、終点から始点にすると 1 対多 / 多対 1 にならないか
- 危険側から安全側ではなく、安全側から危険側にすると BFS しやすくならないか
- 「到達できるか」は逆辺での到達に言い換えられないか

## 6. 頂点を増やすべきではないか

- 頂点ではなく「辺」「向き」「状態」を頂点にした方が自然ではないか
- 盤面問題で方向が効くなら、`(位置, 向き)` を状態に持つべきではないか
- 0/1 コストなら Dijkstra ではなく 01-BFS に落ちないか

## 7. サイクルをそのまま DP していないか

- 依存が循環しているなら、そのまま木 DP / DFS 帰りがけ DP をしてはいけない
- Union-Find, SCC, 同値類圧縮で DAG にできないか
- Functional Graph や permutation のサイクル分解に落ちないか

## 8. 同じ遷移を何回も繰り返していないか

- `K` 回同じ操作なら、ダブリングや行列累乗が疑わしい
- 位置だけでなく、和・最小値・コストも一緒に持てないか
- `K` の表現が 2 進でなくても、10 進や min-plus など別の累乗形にできないか

## 9. クエリは online である必要が本当にあるか

- クエリを先読みできるなら、時系列順に sweep できないか
- 座標圧縮した「点」ではなく「区間」を持つべきではないか
- 毎回構造を更新するより、前から 1 回なめる方が軽くならないか

## 10. 制約の特殊さを使い切っているか

- `n <= 12` や `nCk <= 10^6` のような特殊な制約を見落としていないか
- オーバーフローを無視していないか
- 「大きそうに見えるが実は高々 80 個くらい」みたいな上界を詰めたか

## 11. 期待値・確率なら式を立てたか

- 期待値は「1 回の遷移で何が増えるか」を雑に扱っていないか
- 自己参照する式なら、式変形して `f(n)` を左辺に寄せられないか
- 前向き DP が無理なら、逆向きに「残りの期待値」を定義できないか

## 12. 実装を重くしすぎていないか

- 本質より強いデータ構造を持ち出していないか
- ACL や既存 snippet で済む部分を毎回書き直していないか
- 何を helper に切り出せば、次回 20 行以上減るか

## 最後の 3 手

- 3 ケースだけでいいので小さい例を自分で作る
- その例で「何を固定すると楽か」を日本語で 1 行書く
- それでも動かなければ、`knowledge/` に残す価値のある詰まり方かを確認する
