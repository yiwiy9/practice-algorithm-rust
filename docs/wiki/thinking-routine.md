# Thinking Routine

## 役割

このファイルは、問題を見たときに毎回コメントへ書く思考ルーティンの source of truth とする。

- 問題の考察の糸口を掴む
- 実装可能な状態に落とし込む
- 解法が見えたと思って書き始め、詰めの甘さで手戻りするのを防ぐ
- 近いアルゴリズムの例をすぐ引き、4 問の書き方を即座に真似できるようにする

## コメント用ルーティン

毎回コメントに書く核は、次の 4 問に固定する。

1. 何ごとの話にすると見やすい？ 何だけ持てばいい？
2. それについて、何が分かれば答えになる？
3. 何を捨ててよく、なぜそれで足りる？ 何が効く / 何が禁止？
4. その情報をどう更新 / 判定 / 集計すれば実装できる？

この 4 問を選ぶ理由:

- 1 問目で「見方 / 状態」を決める
- 2 問目で「出口」を決める
- 3 問目で「十分性 / 圧縮 / 効く性質 / 禁止事項」を確認する
- 4 問目で「前計算 / 遷移 / 答えの取り方」を決める

25 分以内に、この 4 問へ各 1 文で答える。
どれか 1 つでも曖昧なら、まだ実装に入らない。

## なぜこの 4 問でほぼ全部に通るか

この 4 問は、別ジャンルの問題にもそのまま当てられる。

- 何ごとの話にすると問題が見やすいか
- その対象について何が分かれば終わるか
- なぜそこまで圧縮して壊れないか、何が効く / 何が禁止か
- それをどうコードに落とすか

つまり 4 問の正体は、

- 見方 / 状態
- 出口 / 欲しい量
- 十分性 / 圧縮 / 禁止事項
- 実装化 / 更新方法

であり、DP / 最短路 / sweep / 座標圧縮 / 検証系 / 数学場合分けを問わず共通している。

## コードコメントにもこの 4 問を書く

このルーティンは頭の中だけで回さず、問題ファイルのコメントにもそのまま書く。

書き方の型:

- 何ごとの話にすると見やすい？ 何だけ持てばいい？
- それについて、何が分かれば答えになる？
- 何を捨ててよく、なぜそれで足りる？ 何が効く / 何が禁止？
- その情報をどう更新 / 判定 / 集計すれば実装できる？

ラベル名は多少変えてよいが、意味は固定する。
重要なのは毎回同じ 4 問に答えること。

## 書き方のコツ

### 1. 1 問目は「何ごとの話にするか」を書く

1 問目には、最終的にコードで持つ状態を書く。
書き方はだいたい

- どこでしか変わらないか
- その間で何が一定か
- だから何だけ持てばよいか

の順で考える。

よくある言い方:

- 境界をまたぐときにしか変わらないので、小立方体ごとの話にする
- `i` を固定すると分類が一定なので、各人を 4 分類だけ持つ
- 作った列全体ではなく、末尾の有効 suffix だけ持つ
- 左端 `l` ごとの話にして、最小成立右端だけ持つ

### 2. 2 問目は「何が分かれば終わるか」を書く

2 問目には、答えへ直結する量を書く。

よくある形:

- 各状態への最小コストが分かればよい
- 各 `l` の最小成立右端が分かればよい
- 各小領域の被覆数が分かればよい
- 各要素の寄与回数が分かればよい
- 候補木が正しいか検証できればよい

ここで「通り数が欲しいのか」「成立境界が欲しいのか」「最小コストが欲しいのか」を取り違えない。

### 3. 3 問目は「なぜそれだけで足りるか」を書く

3 問目には、圧縮して壊れない理由と、必要なら禁止事項を書く。

よくある言い方:

- 一度成立したら以後も成立するので、境界だけ持てばよい
- 被覆数は境界の間で一定なので、元の空間全体は見なくてよい
- 未来は現在位置と残り回数だけで決まり、過去の通り方は不要
- 同じ始点から出る候補は最大のものだけ残せば十分
- 開始後に skip は禁止なので、部分列 DP にはしない

ここが抜けると、「なぜその状態で十分か」が読めなくなる。

### 4. 4 問目は「実装の骨格」を書く

4 問目には、更新 / 判定 / 集計の方法を書く。

よくある形:

- `next_pos` を前計算して各始点から順にジャンプする
- 境界イベントだけ sweep して分類人数を更新する
- 差分配列を作ってセグ木で区間 gcd を取る
- 乗り換え遷移と移動遷移を張って Dijkstra する
- 3 次元 imos + 累積和で被覆数を復元する
- prefix 和で寄与式をまとめて計算する

ここではじめて、実装の main loop や前計算が見える状態にする。

### 5. 手法名は最後に出す

`imos`, `DP`, `二分探索`, `Fenwick Tree` は最後の実装手段であって、本質ではない。

先に書くべきなのは

- 何が変わるか
- 何が一定か
- 何だけ持てば十分か
- 何が禁止か

である。

手法名は、その 4 つが言えた後でよい。

## 良くない書き方と良い書き方

良くない例:

- その情報をどう更新 / 判定 / 集計すれば実装できる？: 座標圧縮して 3 次元 imos 法

これだと「なぜ圧縮してよいか」「何を数えるのか」が抜ける。

良い例:

- 何ごとの話にすると見やすい？ 何だけ持てばいい？
  境界で分割された各小立方体ごとの話にして、各小立方体の被覆数だけ持てばよい。
- それについて、何が分かれば答えになる？
  各小立方体の被覆数が `K` 以上かどうかが分かれば、その体積を足せる。
- 何を捨ててよく、なぜそれで足りる？ 何が効く / 何が禁止？
  元の座標空間全体は見なくてよい。被覆数は直方体の境界をまたぐときにしか変化しないので、
  各小立方体内では一定である。
- その情報をどう更新 / 判定 / 集計すれば実装できる？
  各直方体を圧縮座標上の半開区間に変換し、被覆数を復元して `K` 以上の体積を足す。

## 左端固定 + 最小成立右端 / `next_pos`

### `abc452_d`

- 概要: 部分文字列が `T` を部分列として含む最小右端を数える
- 1. 左端 `l` ごとの話にして、持つのは最小成立右端 `r_min(l)` だけ。
- 2. 各 `l` について `r_min(l)` が分かれば、`r >= r_min(l)` の個数をまとめて数えられる。
- 3. 一度成立したら右端を伸ばしても壊れないので、通り数ではなく境界だけ持てばよい。
- 4. `next_pos[pos][c]` を前計算し、各 `l` から `T` を順に最短マッチさせる。
- 関連: [abc452_d](../../abc452/src/bin/d.rs)

## 二分探索 + 判定

### `joi2014ho3`

- 概要: 円環を 3 区間に切るときの最小区間和の最大化
- 1. 答え候補 `X` を固定した判定問題にして、持つのは各始点から「次に和が `X` 以上になる位置」だけ。
- 2. ある始点から 3 回切って 1 周以内に収まるかが分かればよい。
- 3. `X` が大きいほど判定は厳しくなるので二分探索できる。各区間の中身は持たなくてよい。
- 4. 累積和上で `lower_bound` して次位置を 2 回求め、最後の 1 区間も `X` 以上か判定する。
- 関連: [joi2014ho3](../../kenchon/src/bin/joi2014ho3.rs)

### `abc440_d`

- 概要: `X` 以上の `Y` 番目の欠番を求める
- 1. クエリ `(X, Y)` ごとに、「`A[t]` まで見たとき欠番が何個あるか」の話にする。
- 2. 欠番数が `Y` 以上になる最小の `t` が分かれば答えが決まる。
- 3. `f(t)` は `t` に対して単調増加なので、全部列挙せず境界だけ見ればよい。
- 4. `s = lower_bound(A, X)` を取り、`f(t) >= Y` となる最小 `t` を二分探索して答えへ戻す。
- 関連: [abc440_d](../../abc440/src/bin/d.rs)

## bit 貪欲 / MSB から決める / `DSU`

### `abc408_e`

- 概要: 条件を満たす辺だけで 1 から N へ行ける最小 bitmask
- 1. 答え候補 `x` の話にして、持つのは「この `x` で到達可能か」という判定だけ。
- 2. `w OR x = x` を満たす辺だけで 1 から N に到達できるかが分かればよい。
- 3. 上位 bit から 0 にできるなら 0 にした方が得なので、最小化は bit ごとの貪欲判定に落ちる。
- 4. 上位 bit から 1 本ずつ落としてみて、そのたびに `DSU` で連結判定する。
- 関連: [abc408_e](../../abc408/src/bin/e.rs)

## 尺取り法 / `two pointers` / `sliding window`

### `abc247_e`

- 概要: 区間内の値が `[y, x]` に収まり、`x` と `y` を両方含む区間数
- 1. 現在の window `[left, right]` ごとの話にして、持つのは window の最小値・最大値と `x` / `y` の直近位置。
- 2. 各 `right` に対して、有効な左端が何個あるかが分かれば答えになる。
- 3. 区間条件が壊れたら左端を進めればよく、右追加・左削除の両方が軽いので尺取りできる。
- 4. `right` を伸ばし、壊れている間だけ `left` を進め、`min(last_x, last_y) - left + 1` を足す。
- 関連: [abc247_e](../../kenchon/src/bin/abc247_e.rs), [abc032_c](../../kenchon/src/bin/abc032_c.rs), [abc038_c](../../kenchon/src/bin/abc038_c.rs), [abc130_d](../../kenchon/src/bin/abc130_d.rs), [abc229_d](../../kenchon/src/bin/abc229_d.rs), [abc326_c](../../kenchon/src/bin/abc326_c.rs), [abc381_d](../../kenchon/src/bin/abc381_d.rs), [abc444_e](../../kenchon/src/bin/abc444_e.rs), [arc022_2](../../kenchon/src/bin/arc022_2.rs), [arc098_b](../../kenchon/src/bin/arc098_b.rs), [typical90_ah](../../kenchon/src/bin/typical90_ah.rs)

## Dijkstra / 状態拡張最短路

### `joi2014yo_e`

- 概要: 乗り換えで「あと何本進めるか」が変わる最小コスト移動
- 1. 状態を `(街 i, 残り j 本進める)` に拡張し、位置だけではなく残り回数まで持つ。
- 2. `dist[i][j] = その状態にいる最小コスト` が分かればよい。
- 3. 未来は現在位置と残り回数だけで決まり、過去の通り方は不要。辺重み非負なので Dijkstra でよい。
- 4. 乗り換え遷移と道路移動遷移を張って最短路を回す。
- 関連: [joi2014yo_e](../../kenchon/src/bin/joi2014yo_e.rs)

## 全方位木 DP / `rerooting`

### `abc428_e`

- 概要: 各頂点を根にしたときの最遠距離を全頂点分求める
- 1. 各頂点 `v` ごとに、「子方向の最良値」と「親方向から来る最良値」の話にする。
- 2. 頂点ごとに、全方向を含めた最良値が分かればよい。
- 3. 各根について毎回 DFS し直す必要はなく、子側集約と親側伝播に分ければ 1 回ずつで足りる。
- 4. まず bottom-up で子方向 DP を作り、その後 prefix / suffix を使って親方向 DP を各子へ配る。
- 関連: [abc428_e](../../abc428/src/bin/e.rs)

## `01-BFS` / 向きや辺を頂点化する

### `abc431_e`

- 概要: 盤面上で向きごとの通過コストを最小化する
- 1. マスではなく「そのマスへどの向きで入っているか」を状態にする。

### `joi2012ho4`

## ナップサック DP + 前処理分類

### `joisc2014_m`

- 概要: ストラップ選択で嬉しさを最大化する
- 1. 取り付け順ではなく、現在の空き端子数だけを見る。

- 4. 左から累積和を走査し、`map[S[r] - K]` を加算してから `map[S[r]]` を更新する。
