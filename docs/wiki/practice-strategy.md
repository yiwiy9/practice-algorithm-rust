# Practice Strategy

## このファイルの役割

このファイルは、練習方法とその目的の source of truth とする。

- 何を鍛えるために、どの問題を、何分で区切るかを書く
- 問題ごとの振り返りは別ログに分散させず、各問題のコードコメントに書く
- したがって「運用ルールはここ」「各問題の具体メモは `.rs`」で役割を固定する

## いま解決したい課題

考える力そのものは伸びている。
一方で、

- 25 分で方針の核を出し切れないことがある
- 方針は見えていても、コメントにすると浅くなる
- 実装に入った後は時間制限が緩くなる
- 結果として「最終的には解ける」が「本番速度ではない」

という形になっている。

今のボトルネックは、知識不足そのものより

- contest conversion speed
- 問題の本質を圧縮して言語化する力

の不足である。

したがって、練習では「粘る力」より

- 短時間で方針の芯を出す
- その芯をコメントに落とす
- そのコメントから実装骨格を引き出す

力を優先して鍛える。

## 基本方針

- 本番 25 分以内に方針の核を出す
- 本番 45 分以内に AC まで持っていく
- 「長く考えれば解ける」を「時間内に変換できる」に変える

対象:

- 最近の ABC `E`
- JOI level 7
- 必要に応じて、最近の ABC `D` のうち「固定 / 境界 / クエリ設計」の練習に効くもの

JOI level 7 は、今の感触では最近の ABC `E` より少し軽めか同等帯として扱う。
したがって、まずは最近の ABC `E` と同じ時間ルールに乗せる。

時間ルール:

- `25 分`: 方針 checkpoint
- `45 分`: 総打ち切り

実装に入っていても、`45 分` を超えて粘らない。

## コメントは「圧縮の証拠」

コメントが浅くなる典型は、

- `座標圧縮して imos`
- `DP する`
- `二分探索する`

のように、手法名だけを書いて止まること。

これだと「なぜその手法で十分か」が残らない。
コードコメントに残すべきなのは、手法名そのものではなく、**その手法に落ちる理由**である。

コメントを書くときは、次の順で圧縮する。

1. 問題のルールを言い換える
2. 値や状態が変わる瞬間だけを見る
3. その間で一定な単位に分割する
4. その 1 単位について何が分かれば答えに足せるかを決める
5. その量を出すために必要なクエリ / 状態を決める
6. 単調性・境界・禁止事項で高速化や遷移制約を正当化する

要するに、コメントは

- 何をするか

ではなく

- 何が本質で
- 何を捨ててよく
- 何だけ残せば解けるか

を書く場所である。

## コメント用ルーティン

毎回コメントに書く核は、次の 4 問に固定する。

1. 何ごとの話にすると見やすい？ 何だけ持てばいい？
2. それについて、何が分かれば答えになる？
3. 何を捨ててよく、なぜそれで足りる？ 何が効く / 何が禁止？
4. その情報をどう更新 / 判定 / 集計すれば実装できる？

この 4 問を選ぶ理由:

- 1 問目で「見方 / 状態」を決める
- 2 問目で「出口」を決める
- 3 問目で「十分性 / 圧縮 / 効く性質 / 禁止事項」を確認する
- 4 問目で「前計算 / 遷移 / 答えの取り方」を決める

今の主課題は「見えたと思って実装に入ったが、状態の詰めが甘くて手戻りする」ことなので、
手法名より先に、この 4 問へ答えることをルーティン化する。

25 分以内に、この 4 問へ各 1 文で答える。
どれか 1 つでも曖昧なら、まだ実装に入らない。

## なぜこの 4 問でほぼ全部に通るか

この 4 問は、最近の別ジャンルの問題にもそのまま当てられる。

- `abc452_d`:
  1. 左端 `l` ごとの話にする。
  2. 各 `l` の最小成立右端が分かれば数えられる。
  3. 一度成立したら以後も成立するので、通り数ではなく境界だけで足りる。
  4. `next_pos` を前計算して各 `l` から最小右端を出し、まとめて数える。

- `joi2014yo_e`:
  1. 状態 `(街 i, 残り j 本進める)` の話にする。
  2. その状態にいる最小コストが分かればよい。
  3. 未来の行動は現在位置と残り本数だけで決まり、過去の通り方は不要。
  4. 乗り換え遷移と道路移動遷移を張って Dijkstra する。

- `joi2012ho4`:
  1. 各釘ごとの話にして、最大残り辺長だけ持つ。
  2. その釘が覆われるかどうか、次にどこまで伝播するかが分かる。
  3. 同じ頂点から始まる小さい輪ゴムは最大の輪ゴムに含まれるので捨ててよい。
  4. 下 2 方向へ `x-1` を伝播し、`dp != -1` を数える。

- `joi2013yo_e`:
  1. 境界で区切った小立方体ごとの話にする。
  2. 各小立方体の被覆数が分かれば体積を足せる。
  3. 被覆数は境界をまたぐときにしか変わらないので、各小立方体内では一定。
  4. 各小立方体の被覆数を出して、`K` 以上の体積を足す。

- `abc451_e`:
  1. 各頂点ごとの話にして、親候補だけを持つ。
  2. 候補木が一意に復元できれば、あとは検証できる。
  3. 根からの距離と祖先条件を満たす最も近い頂点だけ見ればよい。
  4. 候補木を復元して、DFS で距離行列を検証する。

- `abc445_e`:
  1. 素数 `p` ごとの話にする。
  2. 各 `A_i` を除いた後の LCM における `p` の指数が分かればよい。
  3. LCM は各素数の指数の最大で決まり、1 位と 2 位だけ見れば足りる。
  4. 素因数分解して各素数の 1 位 / 2 位を管理し、各 `i` の値を合成する。

つまり 4 問の正体は、

- 何ごとの話にすると問題が見やすいか
- その対象について何が分かれば終わるか
- なぜそこまで圧縮して壊れないか、何が効く / 何が禁止か
- それをどうコードに落とすか

であり、DP / 最短路 / sweep / 座標圧縮 / 検証系 / 数学場合分けを問わず共通している。

## コードコメントにもこの 4 問を書く

このルーティンは頭の中だけで回さず、問題ファイルのコメントにもそのまま書く。

書き方の例:

- 何ごとの話にすると見やすい？ 何だけ持てばいい？
- それについて、何が分かれば答えになる？
- 何を捨ててよく、なぜそれで足りる？ 何が効く / 何が禁止？
- その情報をどう更新 / 判定 / 集計すれば実装できる？

ラベル名は多少変えてよいが、意味は固定する。
重要なのは毎回同じ 4 問に答えること。

## 書き方のコツ

### 1. 「どこでしか変わらないか」から書く

問題の本質は、値や状態が変化する場所にあることが多い。

- 境界をまたぐときにしか変わらない
- 文字を 1 つ足したときにしか変わらない
- 操作回数の偶奇が変わるときだけ意味が変わる
- `i -> i+1` のときに高々数個の要素しか状態が変わらない

ここが言えると、「全部を見る必要はない」が出る。

### 2. 「その間で一定なもの」を見つける

変化点が分かったら、その間で何が一定かを見る。

- 境界で分割した小立方体では被覆数が一定
- 固定した `i` では各人の所属可能性は 4 分類で一定
- 今まで作った列全体ではなく、末尾の有効 suffix だけ見れば十分

ここが言えると、「元の大きい対象をそのまま持たなくてよい」が出る。

### 3. 「十分な状態」を 1 段まで圧縮する

元の対象を全部持たず、答えに必要な最小限の状態に落とす。

- 直方体全体ではなく各小立方体の被覆数
- 各人の詳細ではなく `Aのみ / Bのみ / 両方OK / 不可`
- 作った文字列全体ではなく、末尾で列車になりうる suffix の最長長さ

コメント用ルーティンの 1 問目「何ごとの話にすると見やすい？ 何だけ持てばいい？」には、
だいたいこの圧縮後の状態を書く。

### 4. 「何が禁止か」を書く

単調性がない問題でも、禁止事項が芯になることがある。

- 開始後に skip してはいけない
- 操作後に順序が保存される
- 各手では先頭しか取れない

これを書かないと、部分列 DP にしてしまう、余計な遷移を入れる、などの誤読が起きやすい。

### 5. 手法名は最後に出す

`imos`, `DP`, `二分探索`, `Fenwick Tree` は最後の実装手段であって、本質ではない。

先に書くべきなのは

- 何が変わるか
- 何が一定か
- 何だけ持てば十分か
- 何が禁止か

である。

手法名は、その 4 つが言えた後でよい。

## 良くない書き方と良い書き方

良くない例:

- その情報をどう更新 / 判定 / 集計すれば実装できる？: 座標圧縮して 3 次元 imos 法

これだと「なぜ圧縮してよいか」「何を数えるのか」が抜ける。

良い例:

- 何ごとの話にすると見やすい？ 何だけ持てばいい？
  境界で分割された各小立方体ごとの話にして、各小立方体の被覆数だけ持てばよい。
- それについて、何が分かれば答えになる？
  各小立方体の被覆数が `K` 以上かどうかが分かれば、その体積を足せる。
- 何を捨ててよく、なぜそれで足りる？ 何が効く / 何が禁止？
  元の座標空間全体は見なくてよい。被覆数は直方体の境界をまたぐときにしか変化しないので、
  各小立方体内では一定である。
- その情報をどう更新 / 判定 / 集計すれば実装できる？
  各直方体を圧縮座標上の半開区間に変換し、被覆数を復元して `K` 以上の体積を足す。

## 最近の問題の総括

### ABC453 C

- 表面:
  bit 全探索は見えていたが、`0.5` 始まりの扱いで止まった。
- 本質:
  アルゴリズムではなく表現の問題。位置は常に半整数なので、全体を 2 倍すれば整数だけで扱える。
- 言語化:
  「実数に見えるが、分母は固定。符号変化だけ見たいので整数化して十分」と書くべきだった。
- 反省:
  手法以前に、固定分母の整数化を最初に確認する。

### ABC453 E

- 表面:
  `i = 1..N-1`、`N-i`、combination までは見えていたが、その先の分類で止まった。
- 本質:
  固定した `i` に対して各人は `Aのみ / Bのみ / 両方OK / 不可` の 4 通りしかない。
  強制配置を置いたあと、`両方OK` から必要人数を選ぶだけ。
- 言語化:
  「各人を free / forced / forbidden に潰す」「状態が変わるのは境界の高々 4 回だけ」と書くべきだった。
- 反省:
  combination を見る前に、各要素を少数状態に分類できるかを見る。

### JOI2013 本選 2

- 表面:
  prefix を捨てられるので部分列 DP っぽく見えたが、WA になった。
- 本質:
  prefix を捨てられるのは開始前だけで、開始後の skip は禁止。
  さらに必要なのは作った列全体ではなく、末尾にある有効 suffix の最長長さだけ。
- 言語化:
  「開始前と開始後を分ける」「開始後の skip は禁止」「末尾 suffix だけ持てば十分」と書くべきだった。
- 反省:
  単調性がない問題でも、禁止遷移をコメントに書く。

### JOI2013 予選 E

- 表面:
  3 次元 imos は見えたが、コメントが手法名止まりになりがちだった。
- 本質:
  被覆数は直方体の境界をまたぐときにしか変化しない。
  だから境界で空間を分割した小立方体ごとに見れば十分。
- 言語化:
  「どこでしか変わらないか」「その間で何が一定か」「何を足せば答えか」を 3 段で書くべきだった。
- 反省:
  座標圧縮は手法ではなく、一定な単位に空間を切るための道具として書く。

## 1 問の標準プロトコル

1. `0-5 分`: 問題文、制約、サンプル確認
2. `5-10 分`: ルールの言い換えと禁止事項の確認
3. `10-15 分`: 何が変わるか、何が一定かを探す
4. `15-20 分`: コメント用 4 問に 1 文ずつ答える
5. `20-25 分`: 方針を疑似コードか箇条書きにする
6. `25-45 分`: 実装とデバッグ
7. `45 分`: 打ち切って解説を見る

## 45 分で切る理由

- 実装に入った後だけ延長すると、鍛えられるのが speed ではなく endurance になる
- 「方針は合っていたが間に合わない」も、本番では失点なので訓練対象に含める

## 解説を見た後

解説を見たら終わりにしない。

- その場で AC して終わりにしない
- 数時間後か翌日に、`20 分` で白紙再走する
- 再走で詰まるなら、理解ではなく変換速度が足りていないと判断する

`25 分` は、問題文整理・小実験・4 問記述・方針確認までを 1 セットで回す標準時間とする。
慣れてきたら、JOI level 7 など一部は `20/40` に寄せてよい。

## 記録ルール

問題ごとの記録は、別ログではなく各問題のコードコメントに書く。

最低限残すもの:

- コメント用 4 問への答え
- 詰まった理由
- 本質だった変換
- 次に同型が来たら見る checklist / snippet

つまり、練習方法はこのファイルに集約し、
各問題の反省は問題ファイル 1 箇所で見返せるようにする。

## 週次メニューの目安

- 週 2 問: 最近の ABC `E` を `25/45` ルールで解く
- 週 2 問: JOI level 7 を `25/45` ルールで解く
- 週 1 回: 最近解説を見た問題を 1 問だけ `20 分` で白紙再走する
- 週 1 回: `src/lib` に入れる価値のある snippet 候補を見直す

## 今の重点テーマ

- 手法名より先に、変化点・一定な単位・十分な状態を書く
- `通り数` と `成立境界` を取り違えない
- `愚直な見方 -> 十分な状態 -> 更新 / 判定 / 集計方法 -> 前計算` の順で分解する
- online に見える問題を、境界整理や sweep で軽くできるかを疑う
- 実装に入る前に、禁止遷移を言語化する
- fixed denominator の整数化や、巨大状態の圧縮を最初に確認する

## やらないこと

- 最近の ABC `E` を毎回 1 時間以上粘る
- 実装に入ったことを理由に時間制限を外す
- 解説を見て AC しただけで終わる
- 問題ごとの反省を別の場所に散らして、見返し先を増やす
- 観察のあとすぐ「これを DP / 尺取り / データ構造にするには」と型名から入る

## 成功判定

次の状態に近づけば、この方針は機能している。

- 最近の ABC `E` と JOI level 7 で、`25 分` 時点の方針精度が上がる
- `45 分` 以内 AC の本数が増える
- 解説後の `20 分` 再走で、白紙から再現できる問題が増える
- コードコメントに残る反省が、「長く考えた」ではなく「どこで変換できなかったか」になる
- コメントが「手法名」ではなく「なぜその状態で十分か」を説明するようになる
