# Failure Patterns (2026-04-04)

このメモは、「何のアルゴリズムが苦手か」ではなく、
どういう思考の詰まり方をしているかを整理したものです。

## 1. 何を固定するかが遅い

### 症状

- 区間や組をそのまま数えようとして、左右に伸びて止まる
- 条件を満たす範囲が見えても、数え上げに落ちない

### 代表例

- [`abc430/c.rs`](../../abc430/src/bin/c.rs)
- [`abc436/f.rs`](../../abc436/src/bin/f.rs)
- [`abc423/e.rs`](../../abc423/src/bin/e.rs)

### 必要なこと

- 左端、右端、最小要素、最後に使う要素を固定する癖を付ける
- 区間を直接数えず、1 要素の寄与回数で考える癖を付ける

## 2. 状態を増やす発想が遅い

### 症状

- 盤面やグラフを元の頂点のまま解こうとして詰まる
- 辺や向きが本質なのに、頂点だけ見ている

### 代表例

- [`abc429/e.rs`](../../abc429/src/bin/e.rs)
- [`abc431/e.rs`](../../abc431/src/bin/e.rs)
- [`trainings_abc_d_e/abc394_e.rs`](../../trainings_abc_d_e/src/bin/abc394_e.rs)

### 必要なこと

- `(頂点, 向き)`、`辺を頂点化`、`source id を持つ BFS` を手札として常に疑う
- 問題文の状態が頂点 1 個で表せないなら、最初から状態グラフとして書き直す

## 3. サイクルを潰さず DP しようとしてしまう

### 症状

- グラフにサイクルがあるのに、木 DP や帰りがけ順 DP の気分で考え始める
- 「依存がループする」という違和感はあるのに、潰し方がすぐ出てこない

### 代表例

- [`abc435/d.rs`](../../abc435/src/bin/d.rs)
- [`trainings_abc_d_e/abc335_e.rs`](../../trainings_abc_d_e/src/bin/abc335_e.rs)

### 必要なこと

- Union-Find, SCC, 同値類圧縮, 逆辺グラフを早い段階で候補に入れる
- 「この DP の依存先は DAG か」を必ず確認する

## 4. K 回遷移の一般化が弱い

### 症状

- ダブリングは知っているが、位置以外を持つと止まる
- 同じ遷移の繰り返しを、DP や simulation のまま捉えてしまう

### 代表例

- [`abc438/e.rs`](../../abc438/src/bin/e.rs)
- [`abc445/c.rs`](../../abc445/src/bin/c.rs)
- [`abc445/f.rs`](../../abc445/src/bin/f.rs)

### 必要なこと

- `next` だけでなく `aggregate` も一緒に倍化する型を持つ
- 行列累乗は「同じ遷移の高速化」として理解する

## 5. online に見える問題を offline にできていない

### 症状

- 動的データ構造を先に疑ってしまう
- クエリを先読みできるのに、時系列に重く処理しようとする

### 代表例

- [`abc435/e.rs`](../../abc435/src/bin/e.rs)
- [`abc449/e.rs`](../../abc449/src/bin/e.rs)

### 必要なこと

- `query を sort して前から流す` 発想を最初から疑う
- 座標圧縮後に「点」を持つか「区間」を持つかまで考える

## 6. 制約の読み込みと小実験が足りない

### 症状

- 実は小さい上界を見落として重い方針に行く
- 誤読や overflow を後回しにして壊れる

### 代表例

- [`abc386/e.rs`](../../abc386/src/bin/e.rs)
- [`abc430/e.rs`](../../abc430/src/bin/e.rs)
- [`abc445/c.rs`](../../abc445/src/bin/c.rs)
- [`abc450/e.rs`](../../abc450/src/bin/e.rs)

### 必要なこと

- 制約から「何個までしか見なくてよいか」を先に出す
- 複雑すぎると感じたら、誤読を疑う
- 10 行の実験コードや紙の例で見切る

## 7. 期待値・式変形で前向きに考えすぎる

### 症状

- 1 回の遷移で増える量を雑に扱う
- 自己参照の式をそのまま眺めて止まる

### 代表例

- [`abc382/e.rs`](../../abc382/src/bin/e.rs)
- [`abc402/e.rs`](../../abc402/src/bin/e.rs)
- [`abc411/e.rs`](../../abc411/src/bin/e.rs)

### 必要なこと

- 期待値は「何回行うか」を数えているのか「何個増えるか」を数えているのかを分ける
- `f(n)` が右辺にあるなら、まず式変形する
- 前向き遷移が無理なら「残りの期待値」で逆向きに置く

## 8. 実装が毎回フルスクラッチになっている

### 症状

- 方針が分かってからもコード量が多い
- 似た helper を何度も問題ファイルに埋め込んでいる

### 代表例

- 最近のコメント付き `E/F` の行数が長い
- `abc428/e`, `abc430/e`, `abc431/e`, `abc445/f` などで顕著

### 必要なこと

- 2 回以上出た補助実装は `src/lib` 候補として扱う
- code snippet にしないものは wiki の思考 snippet にする

## 総評

あなたの弱点は「難しいアルゴリズムを知らないこと」ではなく、

- 変換を始める観点
- 状態を増やす決断
- online / offline の見切り
- 実装骨格の再利用

の 4 点に集約されます。
ここが整うと、最近の `E` の自力率が一番先に改善するはずです。
